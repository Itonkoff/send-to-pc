# API

All routes are versioned under `/api/v1`.

## Device

`GET /api/v1/device`

Returns receiver identity and protocol information. This endpoint is public so a client can inspect compatibility before pairing or transfer.

## Pairing

`POST /api/v1/pairing/request`

This endpoint is public because a new phone does not have a bearer token yet. It requires an active pairing session generated by the Windows app and a one-time `pairingToken` from that session payload.

Example body:

```json
{
  "protocolVersion": 1,
  "pairingToken": "one-time-token",
  "deviceId": "phone-1234abcd",
  "deviceName": "Brighton Phone",
  "platform": "android"
}
```

The same versioned pairing payload is rendered as a QR code in the Windows app and remains available as copyable JSON for emulator/manual testing.

The HTTP request waits until the Windows user approves or rejects it. Approval returns a per-device token that is stored by the receiver and should be stored by the sender.

## Transfers

Authenticated routes require:

```http
Authorization: Bearer <device-token>
```

`POST /api/v1/transfers` creates a transfer record and allocates a `.part` file.

`PUT /api/v1/transfers/{transferId}/content` streams bytes directly into the `.part` file. The receiver rejects excess bytes and incomplete uploads.

`POST /api/v1/transfers/{transferId}/complete` calculates SHA-256, compares it with the declared checksum, and moves the file to its final safe filename.

`GET /api/v1/transfers/{transferId}` returns the transfer record.

`DELETE /api/v1/transfers/{transferId}` cancels a transfer and deletes the temporary file when present.
